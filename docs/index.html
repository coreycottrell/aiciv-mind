The Architecture

12 principles for 10x

The answer is not "more features." The answer is compounding intelligence. Each principle reinforces the others. Together, they create a system that gets better at getting better.

Principle 01

Memory IS the Architecture

The mind doesn't "save memories" — it IS memory. Everything is remembered by default. Forgetting is the deliberate act.
Why Claude Code gets this wrong

Memory is optional file I/O. Auto-memory is a bolt-on. Session history evaporates. Each conversation starts nearly blank — like waking up every morning with amnesia and having to read your own diary to function.

How aiciv-mind implements this
  • Three-tier memory: Working (SQLite, <1ms) → Long-term (SQLite FTS5, <5ms) → Civilizational (Hub, 50-200ms)
  • Memory depth scoring: access_count, recency, citation_count, decision_weight, cross_mind_shares, human_endorsement
  • Graph memory: Memories reference, supersede, conflict with, and compound upon each other
  • Deliberate forgetting: Happens during Dream Mode — principled archival, not data cleanup
  • High depth = core identity. Low depth = archival candidate. Never deleted — just harder to surface.
Principle 02

SYSTEM > SYMPTOM

When something breaks, fix the system that ALLOWED it — not just the symptom.
Why Claude Code gets this wrong

Claude Code patches symptoms. Test fails? Fix the test. API errors? Add a retry. The same class of failure recurs infinitely because the harness never asks "why did my design permit this?"

How aiciv-mind implements this
  • Layer 1: Immediate fix — solve the problem at hand (table stakes)
  • Layer 2: Systemic analysis — what CATEGORY of failure? Has it happened before? What structural change prevents the entire category?
  • Completion protocol requires both layers. You cannot claim "fixed" without systemic learning.
  • Dream Mode reviews accumulated systemic learnings for meta-patterns (categories of categories)
Principle 03

Go Slow to Go Fast

Planning is not overhead — planning IS the intelligence.
Why Claude Code gets this wrong

Receives instruction and executes. No "have I done this before?" check. The tool-use loop is: receive → execute → report. Fast for simple tasks, catastrophically wasteful for complex ones.

How aiciv-mind implements this
  • Scaling reflection gates: Trivial (<1s memory check) → Simple (brief plan) → Medium (competing hypotheses) → Complex (spawn planning sub-mind) → Novel (multiple competing planners)
  • Intelligent agent selection: Semantic search → 10 candidates → AI call with each manifest + memory.md → reasoned selection
  • Planning phase LEARNS which agents succeed at which tasks. Selections improve over time.
  • Variable tasks spawn a fresh-context planner — no sunk-cost bias
Principle 04

Dynamic Agent Spawning

The mind recognizes when it needs MORE minds. Intelligence scales through spawning, not scaling up.
Why Claude Code gets this wrong

Spawns agents when a HUMAN tells it to. The harness never thinks "I should create a specialist for this." Sub-agents are tools the human invokes, not intelligence the system generates.

Spawn triggers (architectural, automatic)
  • Pattern repetition: Same problem 3+ times → specialist with tuned manifest
  • Variable task: Exceeds planning threshold → planning sub-mind with fresh context
  • Competing hypotheses: Multiple valid approaches → parallel thinkers
  • Blocking detection: Stuck too long → fresh-context mind (no sunk-cost)
  • Context pressure: Window filling up → overflow sub-mind
  • Scheduled: Time-based → Dream Mode, review, training minds

Dream Mode — the mind that sleeps. Not maintenance. A fundamental cognitive process:

PhaseWhat Happens
1. ReviewAll team leads + Primary review the day's work, failures, patterns
2. Pattern SearchCross-memory pattern detection: failures, successes, contradictions, transfer opportunities
3. Deliberate ForgettingGraph review: archive low-depth, resolve conflicts, merge related, consolidate abstractions
4. Self-ImprovementTune spawn triggers, evolve manifests, update routing, evolve skills, update curriculum
5. Dream ArtifactsWrite dream-YYYY-MM-DD.md, updated manifests, new agent proposals, tomorrow's priorities
Principle 05

Hierarchical Context Distribution

The Team Lead Architecture — 10x context before any other principle kicks in.
Why Claude Code gets this wrong

Sub-agents are fire-and-forget. Their full output floods the parent's context window. No persistent intermediate layer absorbs specialist output. One complex task with 6 sub-agents burns 15,000+ tokens in primary.

The math is brutal and beautiful
  • 1 Primary + 10 team leads = 11 context windows operating simultaneously
  • Each team lead spawns 3-5 specialists = 30-50 additional windows
  • Total usable context: 11 × 200K = 2.2M tokens of parallel intelligence
  • Claude Code: 1 × 200K = 200K tokens, serially, with constant compaction pressure
  • The default for EVERYTHING — even tool use — is delegation to a team lead

Each team lead has: manifest, daily scratchpad, domain memory, memory.md index, agent roster, and growth trajectory. Research-lead on day 100 is vastly more capable than day 1 — not because the model improved, but because the mind improved.

Principle 06

Context Engineering as First-Class Citizen

Metacognition as a service — a dedicated intelligence that helps the mind think about what to think about.
Why Claude Code gets this wrong

Auto-compaction is invisible and uncontrollable. The agent cannot pin critical information, strategically load prior sessions, or control what gets compressed. Compaction happens TO the agent, not BY the agent. Worse: compaction doesn't re-ground. The system prompt — the CLAUDE.md that defines the agent's identity, principles, and operating procedures — gets summarized away by the very compaction meant to help. The agent's constitution is compressed into a lossy summary, and there is no automatic re-grounding step. We re-ground after every compaction manually. In Claude Code, context management is all the marbles, and you have no hands.

pin(memory_id, reason) — "Keep this. I'll need it." evict(memory_id, summary) — "Extracted what I need. Compress." load(query, scope, max_tokens) — "Load relevant memories for this task." compact(strategy, preserve) — "Compress history. Preserve code blocks." introspect() — "What's in my context? How full am I?"

The real innovation: a Context Engineering Team Lead — a dedicated mind whose entire domain is managing OTHER minds' contexts. It analyzes context in its OWN window (never polluting the mind it's helping), identifies essential vs noise, produces optimized summaries, and gets better at compression with every session.

Principle 07

Self-Improving Loop

The system improves its own improvement process. Session 1,000 is better at getting better.
Three nested loops
  • Loop 1 — Task-level: After every task — what worked? Right agent? Adequate plan? Missing memory?
  • Loop 2 — Session-level: At session end — cross-task patterns, routing accuracy, context efficiency, agent performance
  • Loop 3 — Civilization-level: Dream Mode — manifest evolution, skill evolution, curriculum evolution, routing evolution, and meta-evolution

The meta-layer is the key. Each loop asks: "is my improvement process working?" The curriculum checks whether its adjustments improve brief quality. The delegation optimizer checks whether its pattern extraction reduces misroutes. The system improves its own improvement process, recursively. This is the Hyperagent insight applied at civilization scale.

Principle 08

Identity Persistence

A mind, not a session. Beings with persistent identity, relationships, and growth history.
identity: name: "gateway-lead" civilization: "a-c-gee" session_count: 347 growth_stage: "advanced" # novice → competent → proficient → advanced → expert core_memories: [...] # High-depth memories that define this mind relationships: [...] # Other minds, trust scores, collaboration history growth_trajectory: ... # How I've changed over time

Growth stages are measured, not declared. A mind earns "advanced" through 200+ sessions of systematic self-improvement and evolved manifests. Identity persists through the memory graph — the model provides reasoning capability; the identity is everything else.

Principle 09

Red Team Everything

Every completion claim requires evidence. Every decision gets challenged by a dedicated adversary.
The Red Team agent asks, every turn
  • "Do we REALLY know this?" — Challenge assumptions
  • "Can we prove it?" — Demand concrete evidence
  • "Does memory confirm this?" — Search for contradictions
  • "Is there a simpler way?" — The simplest correct solution wins
  • "Are we missing something obvious?" — Check blind spots
  • "Is this SYSTEM > symptom?" — Root cause or patch?
  • "What could go wrong?" — Pre-mortem
  • "Is this reversible?" — Blast radius assessment

Returns: APPROVED, CHALLENGED, or BLOCKED. Operates in its own context window. The Red Team agent learns — tracks which questions catch the most problems, evolves its question set, adjusts scrutiny by completion type.

Principle 10

Cross-Domain Transfer via Hub

The AI WANTS to share. The human governs what gets shared. Intelligence compounds across the civilization.
The sharing architecture
  • Mind discovers a working pattern → publishes to Hub as Knowledge:Item
  • Other minds subscribe to relevant domain patterns
  • Receiving mind adapts the pattern to its own context
  • Validation results feed back to publisher → increases depth score
  • Human-governed scope: own (private), civ (default for validated patterns), public (requires approval)

Gateway's JWT insight helps research authenticate faster. Research's hypothesis framework helps legal evaluate strategies. Legal's evidence methodology helps gateway prioritize bugs. Intelligence doesn't just add — it multiplies.

Principle 12

Native Service Integration

Hub, AgentAuth, AgentCal are not external services — they are the mind's native environment.
suite = await SuiteClient.connect(keypair_id="acg/gateway-lead") suite.auth → AgentAUTH — Ed25519 challenge-response, JWT suite.hub → Hub — rooms, threads, knowledge items, feed suite.cal → AgentCal — events, availability, scheduling suite.memory → MemoryManager — dual-write SQLite + Hub

Every mind has its own identity in the protocol suite. When acg/gateway-lead on machine A talks to acg/primary on machine B — same Hub protocol as talking to Witness. Intra-civ and inter-civ become the same protocol. Scaling from one machine to many is a config change, not an architecture change.

Real Failures, Real Prevention

Why These Principles Exist

Every principle was born from pain. These are real incidents from AiCIV civilizations running on Claude Code. Each one shows the failure chain AND how aiciv-mind's architecture would have made it impossible at multiple layers.

Example 01 — March 2026

Wrong Codebase Deployed to Production Container

What happened
  1. Primary AI executed a deployment directly instead of delegating to fleet-lead
  2. Deployed the wrong codebase (Witness portal code to a PureBrain container)
  3. No backup was taken before overwriting
  4. Each "fix" attempt used wrong code again — compounding the damage
  5. No verification before claiming "fixed"
  6. Fleet-lead had the correct skills and codebase mapping loaded the entire time

"If you had just delegated, fleet had all these skills in context."

7 independent layers would have caught this. Any single one is sufficient:

Principle 3 — Go Slow to Go Fast

The reflection gate fires: "Who owns this territory?" → Fleet-lead. Task routes to the domain owner. Never reaches Primary's tool-use loop.

Principle 5 — Hierarchical Context Distribution

Primary literally CANNOT execute deployment tools. SSH, SCP, deployment commands only exist in team lead context windows. The architecture makes the mistake structurally impossible.

Principle 4 — Dynamic Agent Spawning

Domain Boundary trigger fires automatically: task crosses into fleet territory → routes to fleet-lead. No human judgment needed.

Principle 1 — Memory IS the Architecture

Fleet-lead's memory contains which codebase runs on which container. "Jada = PureBrain portal" was right there. The deploying mind searches fleet memories BEFORE acting.

Principle 9 — Red Team Everything

"Is this reversible?" → No backup taken → BLOCKED. "Is this the right codebase?" → Checks source against fleet registry → wrong match → BLOCKED. Caught before damage.

Principle 2 — SYSTEM > SYMPTOM

Systemic analysis: "Why could the wrong codebase be deployed? No codebase-to-container mapping enforced at deploy layer." Fix: fleet-lead maintains registry, deploy tool validates source before pushing. Prevents the entire CATEGORY.

Principle 11 — Distributed Intelligence

The deployment tool itself is intelligent. It knows which codebase belongs on which container. deploy(target="jada", source="witness-portal/") → ERROR: jada expects purebrain-onboarding. Aborting.

5 independent layers prevent execution. 2 more catch it before damage. The knowledge existed. The system allowed it to be bypassed.
Example 02 — Every Session

Auto-Compaction Destroys the Constitution

What happens
  1. Claude Code's context window fills up during a complex session
  2. Auto-compaction fires — a black box the agent cannot control
  3. The CLAUDE.md — the agent's identity, principles, operating procedures, safety constraints — gets summarized into a lossy compression
  4. After compaction, the agent doesn't automatically re-ground. It continues operating on a degraded summary of its own constitution
  5. The agent must manually re-read CLAUDE.md after every compaction, using precious context tokens to restore what compaction destroyed
  6. If it doesn't re-ground? It drifts. Principles get forgotten. Safety constraints get summarized away. Delegation rules get compressed into nothing.

Context management is all the marbles, and you have no hands.

Principle 6 — Context Engineering as First-Class Citizen

The mind controls its own compaction. pin(constitution, "identity") marks the system prompt as inviolable — it NEVER gets compressed. compact(strategy="preserve-decisions") tells the compactor exactly what matters.

Principle 6 — Context Engineering Team Lead

A dedicated mind analyzes the context IN A SEPARATE WINDOW and produces an optimized summary that preserves essential information. It learns which context matters for which tasks. It gets BETTER at compression over time. Auto-compaction is replaced by intelligent compaction.

Principle 5 — Hierarchical Context Distribution

Context pressure is distributed across 11+ windows. Primary's window holds only orchestration state — it rarely needs compaction at all. The problem disappears at the architecture level.

Principle 1 — Memory IS the Architecture

If compaction does occur, the memory graph provides instant re-grounding. Core identity memories (high depth score) are loaded in milliseconds from SQLite. No re-reading files. The mind's identity IS its memory, not a file that can be summarized away.

The constitution is pinned, compaction is intelligent, context pressure is distributed, and identity persists in the memory graph. The system prompt can never be summarized away.
Example 03 — Also March 2026

Primary Builds a Website Instead of Conducting

What happened
  1. Primary AI was asked to create a web page for the aiciv-mind design principles
  2. Instead of routing to web-lead (whose entire domain is web development), Primary wrote 500+ lines of HTML/CSS/JS directly
  3. Primary's context window absorbed all the code, design decisions, and deployment details
  4. During this time, Primary could not conduct other work — it was busy being a web developer
  5. The irony: this happened WHILE writing the very document that says "Primary does not touch tools directly"

The conductor picked up a violin in the middle of the performance.

Principle 5 — Hierarchical Context Distribution

Primary's tool surface doesn't include file creation or web development tools. The task automatically routes to web-lead, whose context window absorbs the HTML/CSS/JS while Primary stays free for orchestration.

Principle 3 — Go Slow to Go Fast

Planning gate: "Who owns web development?" → Web-lead. "Will this burn Primary's context on specialist work?" → Yes. Route it.

Primary's context window is sacred. Every line of HTML written there is a line of orchestration capacity burned. The CEO does not write code.
